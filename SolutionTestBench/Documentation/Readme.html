HP Solution Test Bench Deployment Package
Management Guide

Introduction

The STB Deployment Package is a repository of software and configuration data that you can use to create custom STB Server installers for different user environments. There are portions of this distribution that can be modified to change how STB is installed and configured on a target system. This provides you with the flexibility to enable or disable certain features in STB based on the needs of the user.

The diagram below shows how the creation process is done. RDL will maintain the overall STB deployment respository and will distribute it to you in the form of an installer. You will execute the installer to extract out the STB Deployment Package, which you will then use to create custom STB Server installers (and a STB Client installer) for any of your users. You make your installers unique by changing what plugins will be enabled in the users environment, and by optionally including initial data (test scenarios, printer definitions and test documents) that will be inserted into their database when it is created

This document describes each section of the STB Deployment Package and how to change certain portions to create your own custom installers.

Additional Software Requirements

The STB Deployment Package requires Inno Setup and the Inno Download Plugin to create client and server installers. The specific Inno Setup program used by this distribution is the "Unicode Inno Setup Quickstart Pack". Both of these installers are included in this STB Deployment Package distribution and are located in "Setup\InnoInstallers" for your convenience. They are version 5.5.6 and are known to work with this distribution. A few items to note when installing Inno:

  1. You must install these two packages in a specific order. Install the full Inno Setup package first, followed by the Inno Download Plugin (IDP) second. If you install them in the reverse order then Inno setup will not recognize that the IDP is installed. If you already installed them in reverse order, just install IDP again and that should fix the problem.

  2. When installing the main Inno Setup you can simply accept the default values on each wizard page. The only exception is the page prompting for encryption support. You can disable that feature.

  3. When installing the IDP Setup you can also accept the default values on each wizard page. The only exception is the page prompting for source code download. You can uncheck that if you want.

STB Client and Server Installer Contents

When you build a client or server installer from this STB Deployment Package the following components will be included in each installer..

Client Installer Contents

The client installer contains just the STB User Console which is used by any client workstation within the test lab environment to create, edit and execute STB based tests. The User Console communicates with the STB server to save and retrieve test configurations for edit and execution.

Server Installer Contents

The server installer contains the following components:

  1. The server installer checks the system to determine if Microsoft .NET 4.5.2 Framework is installed and if not it will prompt the administrator to install it. It also includes an option to download the Microsoft .NET 4.5.2 Framework installer from the Microsoft site or if the administrator already has a local copy it allows the local copy to be chosen.

  2. The server installer checks the system to determine if SQL Server Express 2014 is installed and if not it will prompt the administrator to install it. It also includes an option to download the SQL Server Express 2014 installer from the Microsoft site or if the administrator already has a local copy it allows the local copy to be chosen.

  3. The server installer automatically creates a public file share on the server. This shared folder is used to store all test documents, configuration and other components that may be used by any client executing tests within the STB. Note that by default the shared folder is created with full rights for everyone. This configuration assumes the STB server and clients are in a trusted lab environment. If more strict share permissions are required, the administrator will need to adjust them after the installation completes.

  4. After SQL Server Express has been installed, the server installer will create all STB database instances and associated tables. It also inserts some initial data into some of the tables to seed the database properly. There are four different database instances created with SQL Server Express. These four instances match up with the same instances used in the larger scale STF (Scalable Test Framework) that STB is based on. In STF each instance runs on a different server for performance and capacity reasons. In STB, the four instances run on a single server. The four instances are:

    • The Primary STB Configuration Database. This database hosts all test scenarios and configurations.

    • The Asset Inventory Database. This database supports various printer and device definitions used in tests. In addition, STB will use this database to reserve devices to avoid automation conflicts when tests are running.

    • The Test Document Library Database. This database hosts information on various test documents that are stored on the file share described above. These documents can be used by various test automation plugins as needed when performing automation activities such as printing documents.

    • The Report Data Store Database. This database holds all report/result data from executing tests. As tests execute much of the automation will generate runtime data as well as final result data. All this information is stored in this database instance. It can be used for reporting and analysis.

  5. The server installer creates a resident NT service on the server called the STB Data Service. It is a consolidated server that exposes endpoint APIs for the Asset Inventory Database, the Test Document Database, the Report Data Store Database as well as a global Lock Server. The lock service is used by various automation components to ensure multual exclusion when accessing certain resources in the test environment. This is especially important for devices, to ensure that more than one client is not able to send automation commands to the device at the same time.

  6. The server installer contains the STB Admin Control Panel which is a simplified control panel that can be used by the STB administrator configure various aspects of the envirnment. This includes items such as the device inventory (from the Asset Inventory Database), document inventory (from the Test Document Database), server inventory, print server inventory, system settings, enabled automation plugins, user configuration, group configuration, scheduled database cleanup, etc. STB User Console



  7. Like the client installer, the server installer contains the STB User Console which is used by any client workstation within the test lab environment to create, edit and execute STB based tests. The User Console communicates with the STB server to save and retrieve test configurations for edit and execution.

Creating STB Installers

The purpose of the STB Deployment Package is to provide a way for you to create a custom STB Client and Server Installer for each target user. The contents of both installers were listed just above in this document. To create either installer you simply run a utility located in the root of the repository. The STB Server Installer can be customized to provide a different configuration as described below.

Both a client installer and a server installer can be built against the STB Deployment Package by running the CreateInstaller.exe utlity located in the root of the repository. There are two fields that must be filled in before the installer builder will start.

  • Package Label. You can enter anything you choose is this label. It is tied to the installer and presented to the user during the installation process. It can help you later if a user has a problem with their installer as they can give you the version number. The label will also be tied to the actual STB distribution so the user will know what installer was used to install STB.
  • Configuration. This parameter applies primarily to the server installer and tells the builder where to look for any custom configuration data. It is a dropdown list of all the folders resident in the "Configuration" folder off the root of the distribution. The "Default" configuration is included in the initial STB Package Deployment distribution. If you want to create a custom configuration, you can make a copy of the Default folder, name it according to your target user and customize it. More information on what can be configured and how to configure is below.

When the Installer Builder completes there will be new client and/or server installers (based on what you selected) located in the Installers folder off the root of the repository. These are the installers that will be deployed to your target user.

Custom Configuration for Server Installer

From the root of the distribution you will find a "Configuration" folder. This folder contains one or more subfolders that define custom configurations used when creating a server installer. The STB Deployment Package includes a default configuration located in the "Default" folder. It is used by the STB Server Installer creation script if no other target configuration is provided. It is recommended to leave this folder alone and make copies if you want to create configurations that are custom for different users. This way you always have a master copy to reference when needed.

After installation of the STB Deployment Package you will see that the Default configuration folder only contains one file, named "PluginList.txt". The intent of this file and how to used it are described below. In addition to this file, which is required for every additional customer configuration folder you may create, you can also add import files from STF or other instances of STB. The currently supported import files are for test scenarios, print device definitions and test documents. If import files are included the data contained in them will be inserted into the STB database after it is created. More details on the custom import files are below.

Test Automation Plugin Configuration

The "PluginList.txt" file described above contains a list of test automation plugins supported by the system. By default, all plugins are listed in this file, and will be enabled as part of the server installation process if none are eliminated. If you want to create a custom configuration for a target user, as mentioned above, first make a copy of the "Configuration\Default" folder and name it according to your user. Then within the PluginList.txt file you can exclude plugins from being available by commenting them out, or deleting the line from the file if you choose. The example below shows you how plugins can be excluded and ultimately not accessible after the STB server is installed. 

// This is the list of plugins available in STB.  It is a master
// list and can be adjusted for specific distributions.  You can
// simply omit a line by putting one or more "*" characters at the front.
// Note that comment lines in this file are proceeded with "//" and will
// be ignored. 
ActivityOutcome,Activity Outcome Unit Test,Developer
Authentication,Authentication,
CardSwiper,Card Swiper,
CertificateInstaller,CertInstaller,HUB
ClientPullPrintProto,ClientPullPrintProto,
***ConnectivityPrint,Connectivity Print Test,
ControlPanelSecurity,ControlPanel Security Configuration,HUB
ControlPanelValidation,EWS Validation,HUB
***Demo,Demo Plugin,
Email,Exchange Email,
***EpicPrintJob,Epic Print Job,
ePrint,ePrinting,
...

The above example would cause the ConnectivityPrint, Demo and EpicPrintJob plugins from being included in an STB installation.

The format for each line is a comma separated list of three parameters.

  1. The Plugin Name - this must not be changed. If it is then any test that references this plugin will fail.
  2. The Plugin Description - you may change this description if you choose, but it is recommended to only change it if you feel there is something specific about the name you want your target users to know. Otherwise the default name should work in most cases.
  3. The Plugin Group - this defines a category or grouping for the plugin. You will notice that for many lines there is nothing set. If there is a group name defined, then in the editor UI dropdown for plugins there will be a cascading menu item with that group name. This provides a way to group plugins by category. This value should typically not be changes as well, unless there is a need for you to group plugins by a category specific to the needs of your user.

Maintaining Multiple Configurations

As mentioned above, by copying the Default folder within the Configuration folder you can create multiple configurations to support a variety of user installations. Each folder will contain the plugin list you want to enable as well as any import/export files you want to include. The way to enable any particular configuration is to include the name of the folder in the command line of the "CreateServerInstaller.cmd" script. This will tell the installer compiler to look in the specified directory for the configuration information. The section above on creating a Server Installer shows an example of using this parameter to create an installer for "LRS".

STB Import/Export Data Files

As part of your ability to customize an STB Server Installer, you can add import/export files to your custom configuration. These files are created by exporting configuration data from an existing STF or STB distribution. During installation, if you add any of these files to your custom configuration folder (under Configuration in the root of the repository), the installer will attempt to import them into the new or updated server environment. Below is a table that describes the types of import/export files that are supported. Import/export files are XML based.

Name Description
STF/STF Test Scenario Export File (*.stbs) This file contains all the information associated with a full test scenario that has been exported from STF or STB. The data is exported from STF or STB by right-clicking on a scenario from the tree control and selecting Export. When exporting, the user can optionall also choose to include any printer/device definitions as well as embed any test documents used by the test.
Print Device Export File (*.stbp) This file contains all data associated with a print device located in the STF/STB asset inventory database. Print devices are referenced by various plugins that execute print related activities. A print device export file can be created from the STB Admin Console or from the Configuration menu of the STB User Console. It will eventually be integrated into the STF console as well.
Test Document Export File (*.stbd) This file contains all data associated with a test document, including the file itself. When a document is exported, the data is collected as well as the raw data for the file and it is embedded in this export file. A test document export file can be created from the STB Admin Console or from the Configuration menu of the STB User Console. It will eventually be integrated into the STF console as well.

Bringing Up a New Environment

If you are installing the STB Deployment Package for the first time and you do not already have an instance of STB running in your environment, you can use it to build an initial server installer for yourself. Because you don't have a master system already in place to use for source data, after you build and run your server installer the first time your database will be empty except for initial data. It will not have any test scenarios, print device entries or test document entries. You will need to begin adding those manually.

Handling Version Updates

When a new version of STB is released, RDL will release a new version of the STB Package Deployment installer. You will run the installer on your STB packaging server and the from there you can create new installers for your target users. The new installers will include any updates to the STB software as well as any schema changes to the STB databases. Your users will need to run the new server installer your provide them and any clients they have will need to run the client installer to update their environment to the new version.
